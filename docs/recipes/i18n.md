# Recipe: i18n (CZ/EN Bilingual Content)

> **Files:** `frontend/js/public.js` · `backend/database.js` (schema) · `backend/routes/*.js`

---

## 1. How It Works

- Language stored in `localStorage.eolite_lang` → `'cz'` (default) or `'en'`
- DB content: `_cz` / `_en` suffix field pairs (e.g. `title_cz`, `title_en`)
- Frontend helper: `pick(czVal, enVal)` → returns EN if lang=en AND en value is non-empty; else CZ fallback
- Static UI strings: `UI = { cz: {...}, en: {...} }` object in `public.js`
- Language switch re-renders the current view (calls `app.route()`)
- **Admin panel is Czech only — no i18n needed**

---

## 2. DB Schema Convention

```sql
-- Always add _cz and _en pairs for user-facing text
CREATE TABLE IF NOT EXISTS my_table (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  title_cz    TEXT NOT NULL,
  title_en    TEXT,           -- optional, falls back to CZ if empty
  content_cz  TEXT,
  content_en  TEXT,
  -- ...
);
```

---

## 3. Frontend: pick() Helper (public.js)

```javascript
function getLang() {
  return localStorage.getItem('eolite_lang') || 'cz';
}

function pick(cz, en) {
  // Returns EN if lang=en AND en exists (non-empty), else CZ
  const lang = getLang();
  if (lang === 'en' && en) return en;
  return cz;
}

// Usage in render functions:
const title = pick(item.title_cz, item.title_en);
const content = pick(item.content_cz, item.content_en);

// In innerHTML templates:
`<h2>${esc(pick(item.title_cz, item.title_en))}</h2>`
```

// Access:

---

## 5. Language Switch

```javascript
// In public.js — language toggle button handler
document.getElementById('langToggle').addEventListener('click', () => {
  const newLang = getLang() === 'cz' ? 'en' : 'cz';
  localStorage.setItem('eolite_lang', newLang);
  app.route(); // re-render current view
});
```

---

## 6. Backend: What the API Returns

Backend routes return both `_cz` and `_en` fields — the frontend `pick()` decides which to show. The API never language-filters; it always returns complete objects.

```javascript
// routes/magazine.js — returns both fields
router.get('/:slug', async (req, res) => {
  const post = await db.prepare(
    'SELECT id, title_cz, title_en, content_cz, content_en, ... FROM magazine_posts WHERE slug = ? AND is_published = 1'
  ).get(req.params.slug);
  res.json(post);
});
```

```javascript
// public.js — frontend picks the right language
const title   = pick(post.title_cz, post.title_en);
const content = pick(post.content_cz, post.content_en);
```

## 7. Adding a New i18n Field to an Existing Table

DB migration in `backend/database.js`:
```javascript
try { await db.exec(`ALTER TABLE my_table ADD COLUMN subtitle_cz TEXT`); } catch {}
try { await db.exec(`ALTER TABLE my_table ADD COLUMN subtitle_en TEXT`); } catch {}
```

Backend route: include the new columns in SELECT.
Admin form: add two inputs (`subtitleCz`, `subtitleEn`), include in POST/PUT body.
Frontend render: `pick(item.subtitle_cz, item.subtitle_en)`.

## 8. Checklist

- [ ] Every user-facing text field has both `_cz` and `_en` columns in DB
- [ ] `_en` columns are nullable (not `NOT NULL`) — fallback to CZ is expected
- [ ] Frontend uses `pick(czVal, enVal)` — never hardcoded language check outside `pick()`
- [ ] Static labels live in `UI = { cz: {}, en: {} }` in `public.js`
- [ ] Language switch calls `app.route()` to re-render
- [ ] Admin forms have two inputs per bilingual field (CZ required, EN optional)
- [ ] HTML does not contain hardcoded Czech or English text outside of `<script>` data
